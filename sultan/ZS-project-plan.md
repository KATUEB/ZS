# ПЛАН ПРОЕКТА: порт Zombie Survival → s&box

> Это **план** разработки. Не код, не задача к немедленному исполнению — дорожная карта.
> Принцип: **сначала убить технические риски, потом контент.** «Крах» в таких портах = когда наделали
> кучу классов зомби/оружия, а потом выяснили, что физика weld или сеть работают не так. Поэтому
> порядок строго от страшного к простому.

## Источник и лицензия
- Референс дизайна: **Zombie Survival** (JetBoom), GMod, Lua, Source 1.
  https://github.com/JetBoom/zombiesurvival
- Цель: реимплементация геймплея на **C# в s&box** (Source 2, компонентная модель).
- Lua-код НЕ переносится — переосмысляется на компоненты. Репо ZS = дизайн-референс, не источник кода.
- Лицензия JBGM: производные работы разрешены; **монетизация геймплея запрещена** без письменного
  разрешения автора (косметика — можно). Держим некоммерческим по геймплею. Текст лицензии не менять.

## Зафиксированные решения (locked)
- **Сеть:** мультиплеер с самого начала. ВАЖНО: s&box по умолчанию даёт *owner-authority*, НЕ
  server-authority. `[Sync]` сам по себе host-authority НЕ даёт. Host-authority = архитектурная
  дисциплина: не отдавать клиентам ownership критичных объектов, все мутации (покупка, прибивание,
  урон, трата очков) гнать через **Host-RPC с проверкой на хосте**. Иначе клиент-авторитет → тривиальный
  чит (клиент сам себе пишет hp/деньги).
- **v1:** вертикальный срез — 1 класс зомби + 1 оружие + барикады + 1 волна, играбельно end-to-end
  И **«вкусно»** (есть напряжение/фил, не только технически работает). Ширина контента — после.
- **Игроки:** старт **MaxPlayers 24–32**, не 64. Оригинал жил на ~32–40; 64 в s&box с пропами+толпами+
  сетью — серьёзный перф/бандвич-риск. Поднимать по факту замеров, не закладывать 64 как данность.
- **Data-driven с Фазы 2:** оружие/зомби/цены — данные (s&box `GameResource`), не хардкод. Тогда
  Фаза 4 = добавить ассеты данных, а не рефакторить.
- **Канон проекта:** `sweeper/zs/` (единственный с `zs.sbproj`: Title "ZS", Ident "zs", namespace `ZS`,
  GameNetworkType Multiplayer). `sbproj` сейчас MaxPlayers **64 → поправить на 24–32 в Фазе 0**
  (см. решение «Игроки» ниже). Внешний `sweeper/` — мёртвый дефолт-скаффолд.
- **Чистка файлов:** по указанию пользователя файлы пока НЕ трогаем (см. Фаза 0 — отложена).

## Фаза 1 — Технические спайки (де-риск) ⚠️ КРИТИЧНО, БЛОКИРУЮЩАЯ
Доказать, что движок умеет то, на чём держится ZS. Каждый спайк — выкидной прототип, цель да/нет.

ВНИМАНИЕ: S1/S2 переформулированы под реальность s&box (см. ниже) — это НЕ то, что было в черновике.

| Спайк | Вопрос (правильная формулировка) | Если НЕТ → |
|---|---|---|
| **S2 Защита от подделки** (блок., **ПЕРВЫЙ**) | НЕ «синкается ли hp». ## Фаза 2 — Ядро гейммода (костяк)
Минимальный играбельный цикл, host-authoritative с нуля. Без разнообразия контента.
1. **GameMode-менеджер** — состояние матча (warmup → wave → end), главный синглтон-компонент.
2. **Команды:** Survivor / Zombie. По одному классу игрока на сторону.
3. **Раунды/волны** (`logic_waves` ZS → C#): таймеры волн, win/lose (все выжившие мертвы / время вышло).
4. **Смерть → переход в зомби** (фишка ZS).
5. **Барикады** (S1 в продакшн): `Barricadeable` (hp в компоненте) + `Nail` (+hp, макс 4),
   прибивание = kinematic-freeze пропа, hp, разрушение (→dynamic+импульс), ремонт.
6. **Ghosting/фазинг** (core-механика ZS, в черновике её НЕ было): люди медленно проходят сквозь свои
   прибитые пропы, зомби — нет. Коллизии human↔nailed-prop. **НЕ данность — сначала мини-спайк в S4**
   (в Source2 фильтрация по тегам/слоям, предикат на пару может не лечь). Закладывать с барикадами,
   если S4 зелёный.
7. **Плейсхолдер-ассеты** (не ждать Фазы 5): бюджет 1 модель зомби + 1 пушка + 1 проп, чтобы срез
   можно было судить честно. На капсулах-программерарте фил не оценить.

**Выход (= Definition of Done для среза):**
- два клиента (или 1 живой + бот из S6); зомби бьёт барикаду — **хост авторитетен, подделка hp не проходит**;
- выживший чинит молотком; люди фазятся сквозь свой кейд, зомби нет; волна заканчивается по таймеру/смерти;
- **fun-gate:** срез ощущается «вкусно» (паника при пробитии кейда, напряжение волны), не просто
  технически жив. **Требует нескольких зомби-ботов (S6)** — на 1 зомби толпный фил не проверить.

### Ключевая механика барикад (из разбора ZS, переформулировано)
- Ядро: прибитый проп **заморожен** (kinematic, не двигается ничем), hp хранится в компоненте
  (`Barricadeable.Health`), гвозди дают +hp (макс 4). Держится freeze'ом, НЕ джойнтами.
- hp→0 → проп обратно в dynamic + импульс (рассыпается).
- Джойнты (Source2 FixedJoint мягкий/пружинный) — только для много-проповых деплоев, не одиночных пропов.
- Маппинг: `prop_nail`→`Nail:Component`; freeze→`Rigidbody.MotionEnabled=false`/kinematic;
  `baseent.Nails`→`List<Nail>`; мутации hp → Host-RPC+валидация (НЕ голый `[Sync]`).

## Фаза 3 — Экономика и прогрессия
1. **Очки** (`logic_points`): зомби-килл/урон → очки выжившим. Начисление/трата — **на хосте** (часть S2).
2. **Арсенал/магазин:** трата очков на оружие/патроны/деплои. Покупка = Host-RPC, баланс проверяет хост
   (клиент не пишет себе деньги/предметы — иначе чит).
3. **Классы зомби:** разблок по ходу волн (`logic_classunlock`).
4. **Опыт/инфликция** — опционально для v1.

**Выход:** есть смысл играть дольше одной волны.

## Принципы против «краха»
1. **Risk-first:** спайки Фазы 1 обязательны до любого контента.
2. **Вертикальный срез:** сначала 1 оружие + 1 зомби + 1 барикада end-to-end, потом ширина.
3. **Одна система за раз**, каждая компилируется и тестируется.
4. **ZS-репо = референс дизайна, не источник кода.**
5. **Host-authority — дисциплина, не флажок:** критичные мутации через хост, клиент не авторитет.
6. **Data-first с Фазы 2:** контент = данные (`GameResource`), не хардкод.
7. **Fun-gate ≥ technical-gate:** срез судим по ощущению, не только по «движок умеет X».
