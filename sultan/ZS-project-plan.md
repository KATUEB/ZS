# ПЛАН ПРОЕКТА: порт Zombie Survival → s&box

> Это **план** разработки. Не код, не задача к немедленному исполнению — дорожная карта.
> Принцип: **сначала убить технические риски, потом контент.** «Крах» в таких портах = когда наделали
> кучу классов зомби/оружия, а потом выяснили, что физика weld или сеть работают не так. Поэтому
> порядок строго от страшного к простому.

## Источник и лицензия
- Референс дизайна: **Zombie Survival** (JetBoom), GMod, Lua, Source 1.
  https://github.com/JetBoom/zombiesurvival
- Цель: реимплементация геймплея на **C# в s&box** (Source 2, компонентная модель).
- Lua-код НЕ переносится — переосмысляется на компоненты. Репо ZS = дизайн-референс, не источник кода.
- Лицензия JBGM: производные работы разрешены; **монетизация геймплея запрещена** без письменного
  разрешения автора (косметика — можно). Держим некоммерческим по геймплею. Текст лицензии не менять.

## Зафиксированные решения (locked)
- **Сеть:** мультиплеер с самого начала. Все системы — host-authoritative сразу (`[Sync]`, проверки на
  сервере). Никакого «сначала локально, потом прикрутим сеть».
- **v1:** вертикальный срез — 1 класс зомби + 1 оружие + барикады + 1 волна, играбельно end-to-end.
  Ширина контента только после доказанного цикла.
- **Канон проекта:** `sweeper/zs/` (единственный с `zs.sbproj`: Title "ZS", Ident "zs", namespace `ZS`,
  MaxPlayers 64, GameNetworkType Multiplayer). Внешний `sweeper/` — мёртвый дефолт-скаффолд.
- **Чистка файлов:** по указанию пользователя файлы пока НЕ трогаем (см. Фаза 0 — отложена).

## Фаза 0 — Фундамент и чистка (отложена по решению пользователя)
Внешний `sweeper/` — мёртвый дубль (нет `.sbproj`, `.slnx` ссылается на пути вне репо). Канон — `sweeper/zs/`.
Когда дашь добро:
1. Снести дубль: `sweeper/code/`, `sweeper/Assets/`, `sweeper/ProjectSettings/`, `sweeper/sweeper.slnx`.
   Оставить `sweeper/zs/` целиком.
2. (Опц.) поднять `sweeper/zs/` выше — убрать двойную вложенность `ZS/sweeper/zs/`. Churn путей, можно потом.
3. Завести структуру кода в `sweeper/zs/Code/`: `Player/ Barricade/ Rounds/ Economy/ Zombies/ Weapons/ UI/`.
4. Проверить, что `zs.sbproj` открывается и компилится пустой в s&box editor (делает Султан — Claude код не запускает).

**Выход:** чистый компилируемый скелет, геймплея нет.

## Фаза 1 — Технические спайки (де-риск) ⚠️ КРИТИЧНО, БЛОКИРУЮЩАЯ
Доказать, что движок умеет то, на чём держится ZS. Каждый спайк — выкидной прототип, цель да/нет.

| Спайк | Вопрос | Если НЕТ → |
|---|---|---|
| **S1 Weld-физика** (блок.) | `PhysicsJoint`/FixedJoint сваривает 2 пропа? Удаление джойнта → разваливается? | барикады статичные вместо физических |
| **S2 Сеть/авторитет** (блок.) | `[Sync]` + host-authority: hp барикады, команды игроков синкаются между 2 клиентами? | пересмотр MP-модели |
| **S3 Спавн/респавн** | смена команды survivor↔zombie рантайм, респавн на точках? | — |
| **S4 Урон/трейсы** | хитбоксы, `TakeDamage`, melee зомби + ranged оружие? | — |
| **S5 Перф пропов** | заморозка стабильных тел, сколько пропов держит сцена? | лимиты на барикады |

**Правило:** пока S1 и S2 не зелёные — контент НЕ пишем. Это и есть защита от «краха».
**Выход:** документ «движок умеет X, не умеет Y, обходы Z».

## Фаза 2 — Ядро гейммода (костяк)
Минимальный играбельный цикл, host-authoritative с нуля. Без разнообразия контента.
1. **GameMode-менеджер** — состояние матча (warmup → wave → end), главный синглтон-компонент.
2. **Команды:** Survivor / Zombie. По одному классу игрока на сторону.
3. **Раунды/волны** (`logic_waves` ZS → C#): таймеры волн, win/lose (все выжившие мертвы / время вышло).
4. **Смерть → переход в зомби** (фишка ZS).
5. **Барикады** (S1 в продакшн): `Nail` + `Barricadeable`, прибивание пропов, hp, разрушение, ремонт.

**Выход (= Definition of Done для среза):** два реальных клиента; зомби бьёт барикаду — синкается;
выживший чинит молотком; волна заканчивается по таймеру/смерти.

### Ключевая механика барикад (из разбора ZS)
- Барикада = физические пропы, сваренные weld-констрейнтами через «гвозди».
- Hp хранится на пропе (`Barricadeable.Health`), не на гвозде.
- Убей гвозди → weld'ы удаляются (`OnDestroy` → `joint.Remove()`) → конструкция рассыпается физически.
- Больше гвоздей → выше прочность (`RecalculateNailBonuses`).
- Маппинг: `prop_nail`→`Nail:Component`; `constraint.Weld`→`PhysicsJoint/FixedJoint`;
  `baseent.Nails`→`List<Nail>`; netvar→`[Sync]`; `DeleteOnRemove`→`OnDestroy`.

## Фаза 3 — Экономика и прогрессия
1. **Очки** (`logic_points`): зомби-килл/урон → очки выжившим.
2. **Арсенал/магазин:** трата очков на оружие/патроны/деплои.
3. **Классы зомби:** разблок по ходу волн (`logic_classunlock`).
4. **Опыт/инфликция** — опционально для v1.

**Выход:** есть смысл играть дольше одной волны.

---

## Фаза 4 — Контент (масштабируемо, итеративно)
Только когда системы доказаны:
- Оружие выживших (по одному, общий базовый класс).
- Классы зомби (база + наследники).
- Деплои (доски-баррикады, турели, аптечки).
- Боссы.
Каждый — данными/префабом поверх готовых систем, не новый велосипед.
**Выход:** играбельная вертикаль.

## Фаза 5 — Контент-пайплайн и полировка
- Замена ассетов Source1 (модели/звуки/карты в s&box не переносятся) → s&box-аналоги или свои.
- Карты: строить заново в `.vmap` (старые `.bsp` несовместимы).

## Сквозные риски
| Риск | Митигация |
|---|---|
| Физика weld не как в Source1 | Фаза 1 S1 — до контента |
| Сеть/синк барикад | Фаза 1 S2 — до контента; всё host-authoritative |
| Ассеты/карты не переносятся | заложено в Фазу 5, логику не блокирует |
| Лицензия (геймплей-монетизация) | держать некоммерческим, текст LICENSE не менять |
| Дубль-структура проекта | Фаза 0 (отложена по решению пользователя) |
| Раздувание до 50 систем сразу | строгий порядок фаз, спайки выкидные |

## Принципы против «краха»
1. **Risk-first:** спайки Фазы 1 обязательны до любого контента.
2. **Вертикальный срез:** сначала 1 оружие + 1 зомби + 1 барикада end-to-end, потом ширина.
3. **Одна система за раз**, каждая компилируется и тестируется.
4. **ZS-репо = референс дизайна, не источник кода.**
